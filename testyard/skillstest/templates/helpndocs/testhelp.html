Tests & Interviews Page

The Tests & Interviews page enables you to use most of the functionality TestYard has to offer. Specifically, it enables you to:

  • Create New Test
  • Evaluate a Test
  • Access an existing test in edit mode
  • Create an Interview
  • List Tests/Interviews based on Roles
  • Invite a user to take a Test
  • Schedule a Test
  • Schedule an Interview
  • Activate/Deactivate a Test
  • Change Test Visibility
  • Manage Test Invitations
  • Search Tests/Interviews
  • Post a Test on LinkedIn Profile
  • Upload Tests in Bulk
  • Programming Environments

In this help section, we will go over all the above functionalities, and in addition we will also take a look at the interfaces for taking a test and attending an interview. But first, we need to understand a few entities that we would be refering to quite frequently while discussing all this.

Creator/Owner: A Creator or Owner of a test or interview is the user who creates the test or interview. This entity has to be a user who has signed up on TestYard, and has an active account on the platform.
Evaluator: An Evaluator of a test is a user who has been designated as an "Evaluator" by the Test's creator while creating the test. There may be more than one evaluator for a test. Optionally, the creator of the test may also be an evaluator of the same test. This entity has to be a user who has signed up on TestYard, and has an active account on the platform.
Evaluator Group: This entity is a group of evaluators, who have been tagged by a name. So a Test would usually have an Evaluator Group consisting of one or more users on the platform. Since Evaluator Groups are tagged by a name, they may be used in multiple tests. Interviews do not have an evaluator. The user who creates the interview is the one who evaluates the candidate(s).
Candidate: A candidate for a test or interview could be anyone who receives an invitation to attend the same. Candidates need not be a signed up user on TestYard platform. Candidates are identified by their email addresses.
Challenge(s): We use the word "Challenge" and "Question" interchangeably. Basically, these are the questions that your test is going to pose to the test taker.

So now, with terminology out of the way, we can focus on the functional aspects of the application.

Create New Test: The "Test" creation interface has 2 parts to it. Most of the data that needs to be entered in these 2 interfaces have default values set in the interfaces that would, most likely, be good enough for most users. However, to remain as versatile as possible, we thought it would be helpful to offer the users the chance to edit these fields, should they need it.
The two parts of the Test creation interface are the a) Test Metadata interface, and b) The Challenges/Questions interface. When you click on the "Create a Test" button, the overlay interface that appears is the Test Metadata interface. This interface allows you to add/edit the following fields:

    Test Metadata interface:
  • Test Name - mandatory field with no default values. This defines an identifier by which you can later refer to the test you are going to create.
  • Test Type - mandatory field with the default value of "composite". This defines the type of questions/challenges you might add to this test. The available values are self explanatory. The value "composite" allows you to add challenges of all the available types.
  • Rules to Apply - mandatory field with the default value "User may not consult online material during the test". The available rules are: a) Stream video of the test taker during the test, b) Show all challenges/questions at the begining of the test, c) Display challenges/questions to the user one at a time, and d) User may not consult online material during the test. You may select more than one rule, in which case all selected rules will be imposed on the test taker or candidate. However, please note that the options b and c are mutually exclusive. You will get an error if you select both rules and try to create a test.
  • Test Topic - optional field with no default value. This is a categorization field based on the subject matter of the test.
  • Total Score - mandatory field with no default value. Please enter an integer value for this.
  • Number of Challenges - optional field with no default value. You may enter an integer value for this, but it is not imposed while adding challenges/questions. Please leave it blank if you would like to make up your mind on this while adding challenges later.
  • Score Distribution Uniform - boolean field with default value "Yes". This means each of the challenges in your test will have the same weightage. Please select "No" if you want the challenges in your test to have unequal weightage.
  • Negative Scoring Applicable - boolean field with default value "No". Please select "Yes" if you want wrong answers to your challenges to have negative scores.
  • Minimum Pass Score - mandatory field with no default value. Please enter an integer value for this field. It should be less than the Total Score for the test.
  • Existing Evaluator Groups and Group Name fields - mandatory field with no default value. Please refer to the note on Evaluator and Evaluator Group above. If you do not have pre-defined Evaluator Groups available or if you want to create a new Evaluator Group for this test, then please type in the email address(es) of the users who would be the evaluator(s) for your test. Please note that by virtue of you being the creator of this test, you are automatically an evaluator. If you do not wish to be an evaluator for your test, then simply remove your email address from the list of email addresses of your evaluators (in the yellow box).
  • Publish Date - mandatory field with no default value. This is the date from which your test would be available for use. Prior to this date, users (test takers) would see a message stating that the test is not yet available, should they try to access and take the test. Please enter a date in the future.
  • Activation Date - mandatory field with no default value. This is the date from which your test would be activated. Prior to this date, users (test takers) would see a message stating that the test is not yet active, should they try to access and take the test. Please enter a date in the future, and that date should not be a date prior to the "Publish Date".
  • Skill Level - optional field with default value "Intermediate" (if you do not select any of the options, it would be considered as "Intermediate" selection).
  • Test Scope - optional field with default value "private". This determines if your test would be visible to everyone (public), or if it would be visible only to users who are "Connected" with you (protected), or if it would be visible only to you (private). Of course, the test would be accessible to test takers if and when you invite them to take the test, no matter what it's scope is.
  • Answering Language - mandatory field with default value "English-US". You may create a test only with the supported languages listed here.
  • Programming Env. - optional field with default value "None". This determines if a programming environment should be provided by TestYard to the test takers. Currently, only a few languages are offered. Please refer to the note on Programming Environment later in this document.
  • Multimedia Reqd - mandatory field with default value FALSE. This is supposed to determine if the challenges in the test require special multimedia equipment.
  • Random Sequencing - mandatory field with default value FALSE. Please select this checkbox to randomize the sequence of challenges in the test for every test taker.
  • Allow Multiple Attempts - mandatory field with default value FALSE. This determines if this test may be taken by a candidate or test taker multiple times. At present, this has a bug. Please leave this unchecked. Right now you would be able to offer a test taker multiple attempts anyway.
  • Test Duration - mandatory field with no default value. This would be the duration of the test. Please enter an integer value, and select hours or minutes or seconds for the entered integer value.
  • Challenge Duration - mandatory field with default value same as Test Duration. This would be the time allowed for each challenge of the test.
    Challenges interface: The Challenges interface has the following fields.
  • Challenge Statement/Question - This is the statement of the Challenge or question. You may check the "Math Environment" checkbox to access the Math keyboard to type in math expressions. If your browser is Google-Chrome, then you might also use the "Start Recording" and "Stop Recording" buttons to enter the question using Speech Recognition facility of Chrome.
  • Draw figure for this challenge - If your challenge statement needs an accompanying figure or diagram, please click on "click here to draw" to add the figure. You get a very basic drawing utility to add the figure.
  • Upload Media File - If your challenge statement needs an accompanying media file, you may upload it here.
  • External resource URL - If your challenge statement needs an external resource URL, you may add it here.
  • Select Challenge Type - Please select the type of challenge.
  • Max Answer Length - Maximum number of characters allowed for the answer. Default value is 100. Leave empty for no limit.
  • Score for this Challenge/Question - Score for this specific challenge. Please enter an integer value less than the Total Score of this test.
  • Is it compulsory for the candidate to respond? - Leave it unchecked for now.
  • Maximum Time Limit - Maximum time limit (in seconds) allowed for this challenge. The default value is the time limit for th entire test.
  • Quality of the Challenge - How difficult this challenge is. Default value is "Proficient".

That concludes the test creation interfaces. You would be presented these same 2 interfaces when you click on the Edit Interface link in the "Edit Mode Link" column in the "Tests as Creator" listing. Please note that if a test is "Activated", you would only get a read-only view of the test if you click the "Edit Interface" link. To be able to edit the test, you first need to "Deactivate" the test by clicking on the test's "Deactivate Test Now" link. Once the test is deactivated, it would be displayed with a red background. You might need to refresh the screen to update the view.

Evaluate a Test: To be able to evaluate a test, you would first need to bring up the "Tests as Evaluator" listing. This listing will feature tests for which the owner/creator of the test has designated you as one of the evaluators. Once you bring up the "Tests as Evaluator" listing, you will see that each of the tests listed have a "Evaluate" link beside the test's name. Once you click this link, you would be presented with the list of entries, each of which represent a submission from a candidate/test taker. In the "Responses" column of this screen, there would be one of 2 types of links: one with the text "Evaluate Responses", and the other with the text "Show Evaluation".

Clicking on a link with the text "Evaluate Responses" will bring up a popup containing the challenges and the answers from the respective candidate. Each response from the candidate/test taker would have a space for score and a bit of comment. Evaluate each response in this way, and at the end of the popup, you would find a final comment space and a button to save your evaluation. Beside this button, there is a checkbox with the text "Commit Evaluation". Select this checkbox and click on the "Save" button to commit and save your evaluation.

Now if you reopen the evaluation listing again, you would find that the text in the "Responses" column for that candidate has changed to "Show Evaluation". Clicking on this link opens the same popup as before, but this time it would be in read-only mode, since you committed your evaluation in the previous session.

Evaluation is a fairly straight forward process on TestYard. However, should you come across any abnormality, please email with your issue description to support@testyard.in.

Create an Interview:

List Tests/Interviews based on roles: The "Tests & Interviews" page displays 5 role based listing or views - "Tests as Creator", "Tests as Evaluator", "Tests as Candidate", "Interviews Conducted" and "Interviews Attended".

Each of these views lists all tests and interviews that the user is associated with in the 5 roles. These listings are displayed in a tabular format and they provide the interfaces or links to the interfaces (if applicable) for manipulating these tests and interviews.

In the "Tests as Creator" view presents a column named "Completeness". This column displays a score (ideally between 0 and 1) that quantifies how much of each test has been created. Basically, this value is the ratio of the total score of all challenges that have been added to the test divided by the total score of the test as entered by the user while adding metadata for the test.

The interfaces accessible from the "Tests as Creator" view are: a) Invite Candidates Instantly, b) Existing Invitations, c) Edit Mode Link, d) Visibility, e) Schedule Test, f) Activate/Deactivate Test and g) Post on Linkedin. As you might have noticed, there are 2 separate interfaces for inviting candidates for a test. The interface 'a' (Invite Candidates Instantly) should be used if you want to invite one or more candidate(s) for a test, but you do not want to schedule a start time for the test. With this type of invitation, the user would be able to take the test whenever she/he chooses to. The test duration will be imposed on the candidate, but it would allow the candidate to take the test between a valiation period set by the creator. For example, Mike invites John to a test named "Test 1", and sets the validation period from 01-06-2023 00:00:01 to 30-06-2023 11:59:59. If the test duration is 2 hours, then John would be able to take the test any time during the validation period. However, the 2 hour duration rule will be imposed on John and the test will conclude 2 hours after John starts writing the test.

The "Schedule Test" interface allows the creator of the test to schedule a start time (and end time) for every candidate. So, from our earlier example, Mike could schedule John's test to start at 10:00 am on 02-06-2023, and since the test duration is 2 hours, the test would conclude at 12:00 Noon on 02-06-2023.

The "Existing Invitations" interface allow the creator to view the invitations that have been sent out for each of the tests (from either of the 2 scheduling interfaces discussed above). This is a historical view of invitations, but it also allows the creator to activate, deactivate or cancel an existing invitation. Please note that once a test has been taken by the candidate, the deactivation or cancellation of an invitation really doesn't have any significance.

The "Existing Invitations" interface might display 2 tabular structures. These are for "Existing Users" and "Users not Registered on TestYard". Remember that a TestYard test may be taken by anyone with an email address. A candidate/test taker may not be a registered user on TestYard. Hence, we display the invitations information in 2 tabular structures.

The "Schedule Test" interface is quite simple, and consists of a pair of date fields for start and end date/time, and a space of a comma separated list of email addresses. This interface also lists down the schedules along with the email addresses that have been set in the past. If a schedule set in the past has a start time in the future, then such a schedule is editable. So you have the option to update such schedules from this popup view.

In the "Tests as Evaluator" listing, the user is provided with the "Evaluate" interface and the "Post on Linkedin" interface. We have discussed the "Evaluate" interface above. It is a popup based interface. It is a fairly simple interface that allows a evaluator user to evaluate responses from candidates for the given test.

The "Tests as Candidate" listing presents the "Visibility" interface and the "Post on Linkedin" interface only.

The "Visibility" interfaces in "Tests as Creator" and "Tests as Candidates" listings basically allows the user to define the scope of the test's visibility. For example, if I have scored an A+ in a test (as a candidate), then I might want visitors (other TestYard users) to my profile to be able to view it. In such a case, I would set the visibility to "Public". Similarly, for a negative performance, I would want users to not be able to view my failure. In such a case, I might want to set the visibility to "Private". Similarly, if I am a test creator, and I want to be able to certify candidates based on a test I created, then I would want to make its visibility "Public" so that more candidates may take notice of it.

The "Interviews Conducted" listing presents the interfaces "Schedules" and "Post on Linkedin". The "Schedules" interface allows the interview's conductor to schedule interviews. It is a simple interface with a datetime and text field to enter a date & time, and a list of email addresses (separated by commas). Once this is scheduled, both interviewer and the candidate receive email messages specifying the date/time, interview name, and other relevant information and an interview link. Clicking on the link will take both interviewer and candidate to the interview, but the interview interface would be slightly different. Please refer to Create an Interview to learn more about it.

THe "Interviews Attended" listing presents the "Post on Linkedin" interface only. However, it displays the interview link too, so in order to open the interview interface (if somehow you do not receive the email message or lose it somehow), you may simply copy the link and paste it in the browser address bar to open it up.

The "Post on Linkedin" interfaces in each of the listing is a way to post the test/interview specifics on your Linkedin profile.
