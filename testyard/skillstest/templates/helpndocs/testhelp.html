Tests & Interviews Page

The Tests & Interviews page enables you to use most of the functionality TestYard has to offer. Specifically, it enables you to:

  • Create New Test
  • Evaluate a Test
  • Access an existing test in edit mode
  • Create an Interview
  • List Tests/Interviews based on Roles
  • Invite a user to take a Test
  • Schedule a Test
  • Schedule an Interview
  • Activate/Deactivate a Test
  • Change Test Visibility
  • Manage Test Invitations
  • Search Tests/Interviews
  • Post a Test on LinkedIn Profile
  • Upload Tests in Bulk
  • Programming Environments

In this help section, we will go over all the above functionalities, and in addition we will also take a look at the interfaces for taking a test and attending an interview. But first, we need to understand a few entities that we would be refering to quite frequently while discussing all this.

Creator/Owner: A Creator or Owner of a test or interview is the user who creates the test or interview. This entity has to be a user who has signed up on TestYard, and has an active account on the platform.
Evaluator: An Evaluator of a test is a user who has been designated as an "Evaluator" by the Test's creator while creating the test. There may be more than one evaluator for a test. Optionally, the creator of the test may also be an evaluator of the same test. This entity has to be a user who has signed up on TestYard, and has an active account on the platform.
Evaluator Group: This entity is a group of evaluators, who have been tagged by a name. So a Test would usually have an Evaluator Group consisting of one or more users on the platform. Since Evaluator Groups are tagged by a name, they may be used in multiple tests. Interviews do not have an evaluator. The user who creates the interview is the one who evaluates the candidate(s).
Candidate: A candidate for a test or interview could be anyone who receives an invitation to attend the same. Candidates need not be a signed up user on TestYard platform. Candidates are identified by their email addresses.
Challenge(s): We use the word "Challenge" and "Question" interchangeably. Basically, these are the questions that your test is going to pose to the test taker.

So now, with terminology out of the way, we can focus on the functional aspects of the application.

Create New Test: The "Test" creation interface has 2 parts to it. Most of the data that needs to be entered in these 2 interfaces have default values set in the interfaces that would, most likely, be good enough for most users. However, to remain as versatile as possible, we thought it would be helpful to offer the users the chance to edit these fields, should they need it.
The two parts of the Test creation interface are the a) Test Metadata interface, and b) The Challenges/Questions interface. When you click on the "Create a Test" link, the overlay interface that appears is the Test Metadata interface. This interface allows you to add/edit the following fields:

    Test Metadata interface:
  • Test Name - mandatory field with no default values. This defines an identifier by which you can later refer to the test you are going to create.
  • Test Type - mandatory field with the default value of "composite". This defines the type of questions/challenges you might add to this test. The available values are self explanatory. The value "composite" allows you to add challenges of all the available types.
  • Rules to Apply - mandatory field with the default value "User may not consult online material during the test". The available rules are: a) Stream video of the test taker during the test, b) Show all challenges/questions at the begining of the test, c) Display challenges/questions to the user one at a time, and d) User may not consult online material during the test. You may select more than one rule, in which case all selected rules will be imposed on the test taker or candidate. However, please note that the options b and c are mutually exclusive. You will get an error if you select both rules and try to create a test.
  • Test Topic - optional field with no default value. This is a categorization field based on the subject matter of the test.
  • Total Score - mandatory field with no default value. Please enter an integer value for this.
  • Number of Challenges - optional field with no default value. You may enter an integer value for this, but it is not imposed while adding challenges/questions. Please leave it blank if you would like to make up your mind on this while adding challenges later.
  • Score Distribution Uniform - boolean field with default value "Yes". This means each of the challenges in your test will have the same weightage. Please select "No" if you want the challenges in your test to have unequal weightage.
  • Negative Scoring Applicable - boolean field with default value "No". Please select "Yes" if you want wrong answers to your challenges to have negative scores.
  • Minimum Pass Score - mandatory field with no default value. Please enter an integer value for this field. It should be less than the Total Score for the test.
  • Existing Evaluator Groups and Group Name fields - mandatory field with no default value. Please refer to the note on Evaluator and Evaluator Group above. If you do not have pre-defined Evaluator Groups available or if you want to create a new Evaluator Group for this test, then please type in the email address(es) of the users who would be the evaluator(s) for your test. Please note that by virtue of you being the creator of this test, you are automatically an evaluator. If you do not wish to be an evaluator for your test, then simply remove your email address from the list of email addresses of your evaluators (in the yellow box).
  • Publish Date - mandatory field with no default value. This is the date from which your test would be available for use. Prior to this date, users (test takers) would see a message stating that the test is not yet available, should they try to access and take the test. Please enter a date in the future.
  • Activation Date - mandatory field with no default value. This is the date from which your test would be activated. Prior to this date, users (test takers) would see a message stating that the test is not yet active, should they try to access and take the test. Please enter a date in the future, and that date should not be a date prior to the "Publish Date".
  • Skill Level - optional field with default value "Intermediate" (if you do not select any of the options, it would be considered as "Intermediate" selection).
  • Test Scope - optional field with default value "private". This determines if your test would be visible to everyone (public), or if it would be visible only to users who are "Connected" with you (protected), or if it would be visible only to you (private). Of course, the test would be accessible to test takers if and when you invite them to take the test, no matter what it's scope is.
  • Answering Language - mandatory field with default value "English-US". You may create a test only with the supported languages listed here.
  • Programming Env. - optional field with default value "None". This determines if a programming environment should be provided by TestYard to the test takers. Currently, only a few languages are offered. Please refer to the note on Programming Environment later in this document.
  • Multimedia Reqd - mandatory field with default value FALSE. This is supposed to determine if the challenges in the test require special multimedia equipment.
  • Random Sequencing - mandatory field with default value FALSE. Please select this checkbox to randomize the sequence of challenges in the test for every test taker.
  • Allow Multiple Attempts - mandatory field with default value FALSE. This determines if this test may be taken by a candidate or test taker multiple times. At present, this has a bug. Please leave this unchecked. Right now you would be able to offer a test taker multiple attempts anyway.
  • Test Duration - mandatory field with no default value. This would be the duration of the test. Please enter an integer value, and select hours or minutes or seconds for the entered integer value.
  • Challenge Duration - mandatory field with default value same as Test Duration. This would be the time allowed for each challenge of the test.


    Challenges interface: The Challenges interface has the following fields.
  • Challenge Statement/Question - This is the statement of the Challenge or question. You may check the "Math Environment" checkbox to access the Math keyboard to type in math expressions. If your browser is Google-Chrome, then you might also use the "Start Recording" and "Stop Recording" buttons to enter the question using Speech Recognition facility of Chrome.
  • Draw figure for this challenge - If your challenge statement needs an accompanying figure or diagram, please click on "click here to draw" to add the figure. You get a very basic drawing utility to add the figure.
  • Upload Media File - If your challenge statement needs an accompanying media file, you may upload it here.
  • External resource URL - If your challenge statement needs an external resource URL, you may add it here.
  • Select Challenge Type - Please select the type of challenge.
  • Max Answer Length - Maximum number of characters allowed for the answer. Default value is 100. Leave empty for no limit.
  • Score for this Challenge/Question - Score for this specific challenge. Please enter an integer value less than the Total Score of this test.
  • Is it compulsory for the candidate to respond? - Leave it unchecked for now.
  • Maximum Time Limit - Maximum time limit (in seconds) allowed for this challenge. The default value is the time limit for th entire test.
  • Quality of the Challenge - How difficult this challenge is. Default value is "Proficient".

That concludes the test creation interfaces. You would be presented these same 2 interfaces when you click on the Edit Interface link in the "Edit Mode Link" column in the "Tests as Creator" listing. Please note that if a test is "Activated", you would only get a read-only view of the test if you click the "Edit Interface" link. To be able to edit the test, you first need to "Deactivate" the test by clicking on the test's "Deactivate Test Now" link. Once the test is deactivated, it would be displayed with a red background. You might need to refresh the screen to update the view.

Evaluate a Test: To be able to evaluate a test, you would first need to bring up the "Tests as Evaluator" listing. This listing will feature tests for which the owner/creator of the test has designated you as one of the evaluators. Once you bring up the "Tests as Evaluator" listing, you will see that each of the tests listed have a "Evaluate" link beside the test's name. Once you click this link, you would be presented with the list of entries, each of which represent a submission from a candidate/test taker. In the "Responses" column of this screen, there would be one of 2 types of links: one with the text "Evaluate Responses", and the other with the text "Show Evaluation".

Clicking on a link with the text "Evaluate Responses" will bring up a popup containing the challenges and the answers from the respective candidate. Each response from the candidate/test taker would have a space for score and a bit of comment. Evaluate each response in this way, and at the end of the popup, you would find a final comment space and a button to save your evaluation. Beside this button, there is a checkbox with the text "Commit Evaluation". Select this checkbox and click on the "Save" button to commit and save your evaluation.

Now if you reopen the evaluation listing again, you would find that the text in the "Responses" column for that candidate has changed to "Show Evaluation". Clicking on this link opens the same popup as before, but this time it would be in read-only mode, since you committed your evaluation in the previous session.

Evaluation is a fairly straight forward process on TestYard. However, should you come across any abnormality, please email with your issue description to support@testyard.in.

Create an Interview: Clicking on the "Create an Interview" opens the "Enter Interview Metadata" overlay screen. The input entries in this screen are as follows:

  • Interview Title - mandatory field; This should be an alphanumeric text that may be used to uniquely identify this interview.
  • Number of Interviewers - mandatory field; The default value is 1. You may enter an integer value specifying the number of interviewers here.
  • Interviewer Email Ids - mandatory field; By default, the creator's email Id is used. You should still enter the email address in which you would like to receive the interview link for an interviewer.
  • Topic of Interview - mandatory field; No default value. Please enter the topic under which this interview may be classified.
  • Total Score - mandatory field; The default value is 100. Leave it as it is if you do not have any need for a quantifiable score.
  • Candidate should start responding to a question in - mandatory field; But it only applies to automated interviews. Please leave it as it is with the default value of 300 seconds. For automated interviews, this determines how long the interviewer automation would wait for a verbal answer to start.
  • Number of Challenges - mandatory field; The default value is 20. This is valid only for automated interviews. Leave it with the default value of 20.
  • Interview Duration - mandatory field; This determines how long an interview would last. The default value is 1 hour (or 3600 seconds). Please leave it with the default value. This value is not imposed on the interview.
  • Medium - mandatory field; At present only 2 mediums are supported: audiovisual and audio. Please select "Audiovisual".
  • Publish this Interview on - mandatory field; This determines the date from which this interview may be scheduled. Please enter the current/today's date.
  • Allow Realtime - mandatory field; Please check this if you want to conduct the interview. Checking this box displays 3 more fields:
    • Enter Email of Candidate: Mandatory field - please enter the email address(es) of the candidate(s) here. Please note that you may enter one email address only. The field allows multiple email addresses separated by commas, but as of now, interviews with multiple candidates at the same time is NOT supported by the interface that is used to attend the interview.
    • Schedule this Interview on: Please enter the date of the interview.
    • I want to conduct it right now: Please check this if you wish to conduct the interview right now. This sets the date/time for the interview schedule to the current date/time.
  • This Interview is targeted at candidates with - mandatory field; The default value is "Intermediate". Please change as per your need. This value doesn't affect the interview itself. It would be used for statistical purposes only.
  • Interview Scope - mandatory field, with default value "private"; This determines the visibility of the interview. This value doesn't affect the interview session itself.


Clicking on the "Create Interview" button will display a popup screen for the interview session if the "I want to conduct it right now" had been checked. Please refer to the help on "Attend an Interview Session" for more on that interface.

List Tests/Interviews based on roles: The "Tests & Interviews" page displays 5 role based listing or views - "Tests as Creator", "Tests as Evaluator", "Tests as Candidate", "Interviews Conducted" and "Interviews Attended".

Each of these views lists all tests and interviews that the user is associated with in the 5 roles. These listings are displayed in a tabular format and they provide the interfaces or links to the interfaces (if applicable) for manipulating these tests and interviews.

In the "Tests as Creator" view presents a column named "Completeness". This column displays a score (ideally between 0 and 1) that quantifies how much of each test has been created. Basically, this value is the ratio of the total score of all challenges that have been added to the test divided by the total score of the test as entered by the user while adding metadata for the test.

The interfaces accessible from the "Tests as Creator" view are: a) Invite Candidates Instantly, b) Existing Invitations, c) Edit Mode Link, d) Visibility, e) Schedule Test, f) Activate/Deactivate Test and g) Post on Linkedin. As you might have noticed, there are 2 separate interfaces for inviting candidates for a test.

The interface 'a' (Invite Candidates Instantly) should be used if you want to invite one or more candidate(s) for a test, but you do not want to schedule a start time for the test. With this type of invitation, the user would be able to take the test whenever she/he chooses to. The test duration will be imposed on the candidate, but it would allow the candidate to take the test between a valiation period set by the creator. For example, Mike invites John to a test named "Test 1", and sets the validation period from 01-06-2023 00:00:01 to 30-06-2023 11:59:59. If the test duration is 2 hours, then John would be able to take the test any time during the validation period. However, the 2 hour duration rule will be imposed on John and the test will conclude 2 hours after John starts writing the test. The text space for email addresses should contain a comma separated list of email addresses. Please keep the checkbox for "Force Create Fresh URL" checked to ensure that invitation to every email address in the above list contains a fresh test URL. Otherwise, if an invitation exists for an email address from a previously sent invitation for the same test to that email address, then the test URL from that invitation would be reused for the current invitation to the same email address. This could be an issue if you have cancelled or deactivated that URL or if the candidate started taking the test from that invitation but could not complete it for some reason, leaving the test URL unusable.

The "Schedule Test" interface allows the creator of the test to schedule a start time (and end time) for every candidate. So, from our earlier example, Mike could schedule John's test to start at 10:00 am on 02-06-2023, and since the test duration is 2 hours, the test would conclude at 12:00 Noon on 02-06-2023.

The "Schedule Test" interface is quite simple, and consists of a pair of date fields for start and end date/time for the test, and a space of a comma separated list of email addresses. This interface also lists down the schedules along with the email addresses that have been set in the past. If a schedule set in the past has a start time in the future, then such a schedule is editable in the lower section of the interface/screen. So you have the option to update such schedules from this popup view.

In the "Tests as Evaluator" listing, the user is provided with the "Evaluate" interface and the "Post on Linkedin" interface. We have discussed the "Evaluate" interface above. It is a popup based interface. It is a fairly simple interface that allows a evaluator user to evaluate responses from candidates for the given test.

The "Tests as Candidate" listing presents the "Visibility" interface and the "Post on Linkedin" interface only.

The "Visibility" interfaces in "Tests as Creator" and "Tests as Candidates" listings basically allows the user to define the scope of the test's visibility. For example, if I have scored an A+ in a test (as a candidate), then I might want visitors (other TestYard users) to my profile to be able to view it. In such a case, I would set the visibility to "Public". Similarly, for a negative performance, I would want users to not be able to view my failure. In such a case, I might want to set the visibility to "Private". Similarly, if I am a test creator, and I want to be able to certify candidates based on a test I created, then I would want to make its visibility "Public" so that more candidates may take notice of it.

The "Interviews Conducted" listing presents the interfaces "Schedules" and "Post on Linkedin". The "Schedules" interface allows the interview's conductor to schedule interviews. It is a simple interface with a datetime and text field to enter a date & time, and a list of email addresses (separated by commas). Once this is scheduled, both interviewer and the candidate receive email messages specifying the date/time, interview name, and other relevant information and an interview link. Clicking on the link will take both interviewer and candidate to the interview, but the interview interface would be slightly different. Please refer to Create an Interview to learn more about it.

THe "Interviews Attended" listing presents the "Post on Linkedin" interface only. However, it displays the interview link too, so in order to open the interview interface (if somehow you do not receive the email message or lose it somehow), you may simply copy the link and paste it in the browser address bar to open it up.

The "Post on Linkedin" interfaces in each of the listing is a way to post the test/interview specifics on your Linkedin profile.

Schedule an Interview: The creator/owner of an interview may schedule the interview by clicking the "Schedules" button for the interview in the "Interviews Conducted" listing. This would open an overlay screen with rows of existing schedules, and a row for adding more schedules. The user may edit the existing schedules if they have not been conducted yet. At present only the time for the interview may be edited. The list of emails cannot be edited once a schedule has been saved. To create a new schedule, please use the last row that provides a date/time field as well as a space for entering a comma separated list of emails. Once the values have been entered, please click on the "Create Schedule" button for the schedule to be saved. The next time you bring up the "Schedules" overlay screen, you would be able to view your newly created schedule.

Creating a new schedule sends an email to the creator of the interview containing a link to the interview session. It also sends emails to each of the candidate email addresses entered while creating the schedule. Each of these emails also contain a link to the interview session screen. The link sent to the creator is different from the link(s) sent to the candidate(s). Please refer to the help section for attending interview sessions for more on this.

Activate/Deactivate a Test: The "Activate Test Now" and "Deactivate Test Now" link in the "Tests as Creator" listing enables the test's creator to respectively activate and deactivate a test. When a test is created by a user (test creator/owner), the initial state of the test is inactive. To be able to take the test by a test taker, the test has to be activated by the creator. This may be done by clicking this link. The "Deactivate Test Now" is also necessary since if the creator needs to edit the test for some reason, it has to be deactivated first since an active test cannot be edited. Editing a test may be done for various reasons like change of weightage in the challenges/questions, change or addition of new challenges/questions, etc. This interface allows you to perform these actions.

Change Test Visibility: The checkboxes of this column allows the user to change the visibility of the test. The visibility column appears in 2 listings: "Tests as Creator" and "Tests as Candidate", and it means different things in each of them. From the test creator's/owner's perspective, the visibility of the test determines if the test is visible to other users (could be connections or from test groups) as a "test" entity. From the candidate's perspective, visibility of the test determines if the performance of the candidate in the tests may be viewed by other users. So for example, if John takes Mike's test (test created by Mike) and John want's to keep the performance private, then others would not be able to view the score of John in that test. The test may, however, be publicly visible as a test entity if Mike wants to keep the visibility "public".

The "protected" visibility status ensures that the test (either the test entity or the candidate's performance in it) is visible only to "connections" the user has accepted in the "Network" page of TestYard. Please refer to "Network" pages help for more on connections and groups on TestYard.

Manage Test Invitations:The "Manage Invitations" links in the "Existing Invitations" column allow the creator to view the invitations that have been sent out for each of the tests (from any of the 2 scheduling interfaces discussed above). This is a historical view of invitations, and it also allows the creator to activate, deactivate or cancel an existing invitation. Please note that once a test has been taken by the candidate, the deactivation or cancellation of an invitation really doesn't have any significance.

The cancellation of an invitation ensures that the URL of the test for the associated candidate is made unusable permanently. If the candidate is to take the test after cancellation of the invitation by the test's creator/owner, then the candidate must request the creator to send a new invitation for the test.

The "Existing Invitations" interface might display 2 tabular structures. These are for "Existing Users" and "Users not Registered on TestYard". Remember that a TestYard test may be taken by anyone with an email address. A candidate/test-taker may not be a registered user on TestYard. Hence, we display the invitations information for both types of users in the 2 tabular structures.

Search Tests/Interviews: The "Search" bar at the top of "Tests & Interviews" listings allow the user to search for tests and interviews with respect to a role. Users may also search tests and interviews from the "Search" page. However, the search feature in this page (Tests & Interviews) allows the user to search tests/interviews with respect to a role. So, if you are using this search feature from the "Tests as Evaluator" listing, then it will only look up tests in which the user is an evaluator. This is a convenient way to look up tests/interviews if you have a lot of tests and interviews on TestYard, and your listings run multiple pages.

Post a Test on Linkedin Profile: This feature simply allows the user to post a notification about a test on the user's Linkedin profile. Exactly how the notification looks will depend on the role the user has with respect to the test. For example, if the user is the creator of the test, then the notification would say that the user has created this test, and call out users visiting the Linkedin profile to take the test. From a candidate's perspective, this notification would display the performance of the user in the test.

Upload Tests in Bulk: This is an experimental interface to allow users to create tests in bulk. This feature is not available for interviews (actually it makes no sense for interviews). The user may upload upto 10 tests at a time. The formats supported are csv, xml and xls(x). You will find the structure of these formats from the interface itself by clicking the download link for the template of the tests. More help is available from the interface in the form of a text file which you may download from the link for help text for the template file. Also, sample tests in csv, xml and xls are available from the interface.

The use case for the scenario in which you might use this feature is a terminal examination of a formal school or an educational institution. Such examinations involve separate tests in multiple subjects over a short period (for example, a week). The creator may generate the xml or csv or xls for each test by downloading the sample test in the desired format and replacing the entries in it with appropriate entries for the tests. Finally, she may upload them at once, and then simply activate the tests in the "Tests as Creator" listing.

Programming Environments: As of now, TestYard provides a very limited set of programming environments that you may choose from to use in a test. This feature is solely for the perusal by the software technology fraternity, and hence it has only limited use. That is the reason why it has not been implemented with all major programming environments. However, this will definitely not be the case in future. Currently, this feature is being worked on actively, and there would be major additions to this feature within the year 2024. The environments available as of now are as follows:

  • Python 3.x
  • Perl 5.x
  • Bash
  • Ruby
  • Javascript/TypeScript

As you might notice, only a few interpreted languages are supported right now. We intend to extend support for the major compiled languages like C/C++, Java and Rust in the near future.

Please update me when this happens!

Attending an Interview Session: An interview session screen opens up as a popup in one of the 2 different ways - you have clicked the "Create Interview" button in the "Enter Interview Metadata" screen, or you have clicked a link for an interview (either as a interview conductor or as an candidate) in your email message from a TestYard user. Either way, you end up with the interview session popup. Let us take a good look at what is available for the user in that popup screen.

The popup screen will differ very slightly, based on whether you are the interview conductor or a interview candidate. However, for the most part, it would look the same. We will go over the part that remains same for both roles here, and then we will discuss the difference. Here are the common fields and controls:

  • Screen Name - You need to enter an alphanumeric string that identifies yourself on the peer's interview screen. Your firstname would be a good choice to use here.
  • Share Screen button - Once the interview starts, you can click on this button to share your screen's view.
  • Start Recording button - Click on this if you want to record the call.
Below these controls, you will have the area where you would see the view from the peer's webcam. In the inset, you will have the view from your webcam. Below this, you will find the 2 checkboxes for enabling/disabling the webcam and microphone.
If you go further down, you will find the chat area where you can have a text chat with your peer. It also supports sending files both ways. The maximum file size supported now is 5 MB.

The difference between the interview conductor and interview candidate/attendee view is the button on the right hand corner of the video display area. For an interview conductor, this button would be "Start", while for the interview candidate/attendee it would be named "Join". So the interview conductor has the privilege to start the session, while the candidate can join such an interview. Also, the candidate would require permission from the conductor to join the interview. This happens when the candidate clicks the "Join" button. An intimation is sent to the interview conductor asking her/him to allow the candidate to join in the call.

Note: if the interview popup window opens up as fullscreen, please resize it so that your webcam view appears as an inset in the peer's webcam view.
